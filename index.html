Section 01

Two-Agent Architecture with Separation of Concerns

The system is deliberately split across two AI agents with non-overlapping responsibilities. This isn't a UX decision — it's an architectural one. Each agent operates in the context it's best suited for, with shared state managed through a persistent file layer in Google Drive rather than through conversation memory.

This eliminates a core failure mode in most AI workflows: context drift. Because neither agent holds truth in memory, both agents always read fresh state from file. A context reset never loses data — it's already in the files.

Agent A

Claude Chat — Real-Time Layer

  • Daily nutrition tracking (mobile, on-the-go)
  • Live meal decisions & macro calculations
  • Single-card generation mid-week
  • Daily handoff document generation
  • Quick training or nutrition Q&A
Agent B

Cowork — Batch Processing Layer

  • Weekly workout card generation (all 4 cards)
  • CSV ingestion & grooming pipeline
  • Body composition trend analysis
  • Cardio session photo processing
  • Project file editing & versioning
The split is driven by a real constraint: food tracking happens on mobile in real-time, while batch data processing requires desktop. Designing around actual user behavior — not ideal behavior — is what makes a system durable.
Section 02

ETL Data Pipeline for Training Intelligence

Workout card generation isn't a single LLM call — it's a structured pipeline. Raw exports from the RepCount fitness app are ingested, filtered to the current mesocycle, compressed into a lean history file, and then consumed by the card generation skill. This keeps context lean, avoids token bloat from irrelevant history, and ensures the model reasons over clean, scoped data.

Edge cases are explicitly handled in the pipeline spec: non-breaking spaces in CSV timestamps require UTF-8 encoding overrides; CSV export dates are validated against the current date before rotation logic is applied; multiple same-day scale readings use only the first (fasted morning) entry. These aren't afterthoughts — they're catalogued constraints the system enforces on every run.

🔁

Cardio Photo Pipeline

Polar/Fitbit session screenshots are dropped into a watched folder. The cardio skill reads the image, extracts session metrics, assesses Active Rest Day eligibility, logs to the tracking file, then deletes the source image — all in one automated pass.

📊

Body Comp Analysis

Smart scale CSV is parsed for 7-day rolling average weight, weekly low floor (stall detection), and Fat-Free Body Weight trend. Multiple signals are used because any single metric can mislead — the system is designed to interpret the full picture.

Section 04

Prompt Engineering at Scale

The system is governed by five versioned instruction files that act as the single source of truth across all agents and sessions. Changes are surgical — a specific exercise prescription or nutrition rule is edited in one file, and every future interaction picks it up automatically. There's no re-prompting users, no chat context to maintain.

# Project file structure — each file has a defined scope and version
training_instructions_v3.8.txt # Workout programming, deload criteria, exercise aliases coach_instructions_v3.9.txt # Nutrition protocols, flex week scheduling food_database_v2.10.txt # Macro database — always read before any food response active_flags.txt # Live watch items — regenerated every week with cards project_instructions_v1.20.txt # System behavior, routing rules, agent contracts

Every file edit follows a strict protocol: read current → archive old version → write updated version. This creates a full audit trail. The system knows what it knew at any point in time.

🗺️

Routing Decision Tree

Every type of user request maps to an explicit file-read sequence. "Generate week X cards" has a different read chain than "what should I eat." No guessing about what context is needed.

🔖

Exercise Alias Mapping

The app names exercises differently than the instruction files. A dedicated alias table reconciles them on every CSV parse — preventing silent mismatches from corrupting progression data.

⚡

Context Window Budget

The grooming skill exists specifically to compress full CSV history into a current-meso view. This keeps token usage lean and prevents the model from reasoning over irrelevant historical data.

Section 05

Self-Healing Error Architecture

Most AI systems treat errors as one-time fixes. This system treats them as system-level defects that require root cause classification, correction, and permanent inoculation. Every error goes through a formal protocol before it's considered resolved.

ERROR RESOLUTION PROTOCOL

Step 1: Clarify — what happened, what should have happened, what does "solved" look like Step 2: Classify — one of 8 error types (Data Misread / Logic Error / Template Misapply / Stale Assumption / Ambiguous Rule / Missing Rule / Scope Creep / Wrong Format) Step 3: Fix — restate corrected output Step 4: Auto-Reset — edit relevant project file directly Step 5: Inoculate — add to Known Pitfalls log ← error cannot recur without triggering this entry

The Known Pitfalls log now has 16 entries — each one a specific, dated failure mode with an explicit prevention rule. Before every workout card generation, the system is instructed to scan this log. It's a regression test suite for prompts.

Warm-up Set Misclassification

Early-sequence sets were labeled as warm-ups based on position. Rule: check RIR. Any set ≤4 RIR is a working set, regardless of where it appears in the sequence.

CSV Set Count Override

Program template counts were overriding actual CSV data. Rule: CSV is ground truth. When historical sessions consistently show N sets, prescribe N sets — never defer to the template.

Mid-Rotation Deload Declaration

A deload status note was treated as a trigger mid-rotation. Rule: deload assessment fires ONLY after Limbs B closes a complete rotation. Any mid-rotation deload declaration is always an error.

Section 06

Event-Driven Automation & Scheduling

The system runs on four recurring automated workflows, triggered via Google Calendar events. Each calendar event contains a copy-paste prompt in its description — so execution is consistent, repeatable, and doesn't depend on the user remembering what to do or how to phrase it.

🏋️

Weekly Check-In & Cards

Every Sunday at 7pm ET. Runs CSV grooming, rolling average calculation, flag review, card generation for the upcoming week, and food-log archiving. One prompt, full pipeline.

📸

Daily Cardio Log

Every day at 9pm ET. Checks the watched folder for new cardio screenshots, processes and logs any found, assesses Active Rest Day eligibility, deletes source images.

🔍

Biweekly System Audit

Every other Sunday. Reviews 14-day progression trends, flag accumulation patterns, body comp direction, and system health. Ends with one concrete recommendation.

🗂️

Quarterly Archive Review

Quarterly on the 2nd Sunday of Jan/Apr/Jul/Oct. Reviews all archive folders, advises on retention — keeping the system clean without losing historical data.

The calendar prompt pattern means the user never has to remember workflow steps or re-derive what to say. Consistency of execution is a feature, not an expectation.
Section 07

Context Management at Scale

Long-running AI systems accumulate drift — the model's understanding of "current state" diverges from reality as conversations grow longer. This system solves that architecturally rather than through careful prompting.

🧼

Chat Hygiene Protocol

Explicit rules govern when to start a new chat: new training week, new CSV, next-day food tracking, any compaction event. Each chat is stateless by design — project files carry all context, not conversation history.

📋

Active Flags File

A plain-text watch-item file is regenerated alongside every weekly batch of workout cards. It's read at the start of every training interaction. Live system state is always one file-read away.

📎

Daily Handoff Protocol

At the end of each food tracking day in Chat, a compact handoff block is generated. Downloaded as PDF → uploaded to Drive → available to Cowork for trend analysis. State persists across both agents without requiring shared memory.

⚠️

Session Length Monitoring

After 3+ major file reads or edits in a Cowork session, the system proactively warns that context is accumulating and recommends wrapping up. Drift prevention is explicit, not passive.

Section 08

Product Thinking Applied to AI Systems

The technical architecture reflects a set of deliberate product decisions — constraints observed, tradeoffs made, and user behavior respected rather than corrected.

📱

Design Around Real Behavior

Food tracking stays in Chat — not because it's ideal, but because tracking happens on mobile in real time. The system accommodates this constraint explicitly rather than asking the user to change their behavior.

📈

Data Overrides Generic Advice

Training prescriptions are derived from personal historical CSV data, not population norms. The system is explicitly instructed to use the user's own dose-response history over general bodybuilding guidelines.

🔬

Science Claim Verification

Any physiology, nutrition science, or training science claim requires a web search and real citation before being surfaced. The system never asserts from memory on empirical questions.

🌱

Progressive Enhancement

The system was built incrementally — core coaching first, then skills, then automation, then audit cadences. Each layer was added only when a real recurring need was demonstrated, not anticipated.

🔎

Skill Migration Detection

During biweekly audits, the system actively scans for manual workflows that have calcified into recurring habits and flags them as automation candidates. The system improves itself over time.

🚦

Structural Change Gate

Before any new skill or workflow is proposed, a 3-point check runs: fits token budget? manually triggered? solves a real recurring problem? All three must pass. No speculative complexity.
