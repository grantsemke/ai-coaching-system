Section 01

Two-Agent Architecture with Separation of Concerns

I split the system across two AI agents with non-overlapping responsibilities. It's not a UX choice — it's an architectural one. Each agent does what it's best suited for, and they share state through a persistent file layer in Google Drive instead of relying on conversation memory.

This solves one of the biggest failure modes in AI workflows: context drift. Neither agent holds truth in memory — they always read fresh state from file. If a session resets, nothing is lost. It was already in the files.

Agent A

Claude Chat — Real-Time Layer

  • Daily nutrition tracking (mobile, on-the-go)
  • Live meal decisions & macro calculations
  • Single-card generation mid-week
  • Daily handoff document generation
  • Quick training or nutrition Q&A
Agent B

Cowork — Batch Processing Layer

  • Weekly workout card generation (all 4 cards)
  • CSV ingestion & grooming pipeline
  • Body composition trend analysis
  • Cardio session photo processing
  • Project file editing & versioning
The split came from a real constraint — I track food on my phone in real time, but batch processing needs a desktop. I built around how I actually behave, not how I'd ideally behave. That's what makes it stick.
Section 02

ETL Data Pipeline for Training Intelligence

Generating my weekly workout cards isn't a single AI call — it's a structured pipeline. Raw exports from my workout tracking app get ingested, filtered down to just the current training block, compressed into a lean history file, and then fed into the card generation skill. This keeps the context lean and token-efficient, and ensures the model is reasoning over clean, relevant data — not months of history it doesn't need.

Cardio Photo Pipeline

I drop my Fitbit session screenshots into a watched folder. The cardio skill reads the image, extracts session data, checks if I qualify for a higher calorie target that day, logs everything, then deletes the photo — all in one pass.

Body Comp Analysis

My smart scale CSV gets parsed for a 7-day rolling average, the weekly low floor, and my fat-free body weight trend. I track three signals because any single one can mislead — the system is built to interpret the full picture.

Section 04

Prompt Engineering at Scale

Five versioned instruction files are the single source of truth across both agents and every session. When something changes — a new exercise, an updated nutrition rule — I edit one file and every future interaction picks it up automatically. No re-prompting, no chat context to maintain.

# Project file structure — each file has a defined scope and version
training_instructions_v3.8.txt # Workout programming, deload criteria, exercise aliases coach_instructions_v3.9.txt # Nutrition protocols, flex week scheduling food_database_v2.10.txt # Macro database — always read before any food response active_flags.txt # Live watch items — regenerated every week with cards project_instructions_v1.20.txt # System behavior, routing rules, agent contracts

Every edit follows the same protocol: read current → archive the old version → write the update. I have a full audit trail and can see exactly what the system knew at any point in time.

🗺️

Routing Decision Tree

Every type of request maps to an explicit file-read sequence. "Generate week X cards" has a different read chain than "what should I eat." No guessing about what context is needed.

🔖

Exercise Alias Mapping

My tracking app names exercises differently than the instruction files. A dedicated alias table reconciles them on every CSV parse — preventing silent mismatches from corrupting progression data.

⚡

Context Window Budget

The grooming skill exists specifically to compress full CSV history into a current-meso view. This keeps token usage lean and prevents the model from reasoning over irrelevant historical data.

Section 05

Self-Healing Error Architecture

Most people just fix the prompt when something goes wrong. I treat errors as system-level defects — they get classified, corrected, and permanently inoculated against. Every error goes through a formal protocol before it's considered resolved.

ERROR RESOLUTION PROTOCOL

Step 1: Clarify — what happened, what should have happened, what does "solved" look like Step 2: Classify — one of 8 error types (Data Misread / Logic Error / Template Misapply / Stale Assumption / Ambiguous Rule / Missing Rule / Scope Creep / Wrong Format) Step 3: Fix — restate corrected output Step 4: Auto-Reset — edit relevant project file directly Step 5: Inoculate — add to Known Pitfalls log ← error cannot recur without triggering this entry

The Known Pitfalls log has 13 entries — each one a specific, dated failure with an explicit rule to prevent it from happening again. Before every workout card gets generated, I scan that log. It's essentially a regression test suite for prompts.

Warm-up Set Misclassification

Early-sequence sets were labeled as warm-ups based on position. Rule: check RIR. Any set ≤4 RIR is a working set, regardless of where it appears in the sequence.

CSV Set Count Override

Program template counts were overriding actual CSV data. Rule: CSV is ground truth. When historical sessions consistently show N sets, prescribe N sets — never defer to the template.

Mid-Rotation Deload Declaration

A deload status note was treated as a trigger mid-rotation. Rule: deload assessment fires ONLY after Limbs B closes a complete rotation. Any mid-rotation deload declaration is always an error.

Section 06

Event-Driven Automation & Scheduling

I run four recurring workflows triggered by Google Calendar events. Each event has a copy-paste prompt in its description — so I never have to remember what to say or how to kick things off. Consistency of execution is built in, not hoped for.

🏋️

Weekly Check-In & Cards

Every Sunday at 7pm ET. Runs CSV grooming, rolling average calculation, flag review, card generation for the upcoming week, and food-log archiving. One prompt, full pipeline.

📸

Daily Cardio Log

Every day at 9pm ET. Checks the watched folder for new cardio screenshots, processes and logs any found, assesses Active Rest Day eligibility, deletes source images.

🔍

Biweekly System Audit

Every other Sunday. Reviews 14-day progression trends, flag accumulation patterns, body comp direction, and system health. Ends with one concrete recommendation.

🗂️

Quarterly Archive Review

Quarterly on the 2nd Sunday of Jan/Apr/Jul/Oct. Reviews all archive folders, advises on retention — keeping the system clean without losing historical data.

The calendar reminds me, and the prompt tells me exactly what to paste. I don't have to remember the steps or re-derive what to say. Showing up is the only thing I have to figure out.
Section 07

Context Management at Scale

The longer an AI conversation runs, the more its sense of "current state" drifts from reality. I solved that architecturally — not by trying to write better prompts.

🧼

Chat Hygiene Protocol

Explicit rules govern when to start a new chat: new training week, new CSV, next-day food tracking, any compaction event. Each chat is stateless by design — project files carry all context, not conversation history.

📋

Active Flags File

A plain-text watch-item file is regenerated alongside every weekly batch of workout cards. It's read at the start of every training interaction. Live system state is always one file-read away.

📎

Daily Handoff Protocol

At the end of each food tracking day, I generate a compact handoff block. Downloaded as PDF → uploaded to Drive → available for trend analysis. State persists across both agents without requiring shared memory.

⚠️

Session Length Monitoring

After 3+ major file reads or edits in a session, the system proactively warns that context is accumulating and recommends wrapping up. Drift prevention is explicit, not passive.

Section 08

Product Thinking Applied to AI Systems

I built this incrementally and for myself — and I think that shows. Every decision came from a real constraint I ran into, a tradeoff I had to make, or a behavior I designed around instead of fought against. I manage my fitness through both food and activity, and the system reflects exactly how I actually operate.

📱

Design Around Real Behavior

Food tracking stays in Chat — not because it's ideal, but because I track on my phone in real time. I built around that constraint instead of trying to fight it.

📈

My Data Overrides Generic Advice

My training prescriptions come from my own historical data, not population norms. I tell it to use my personal dose-response history over general bodybuilding guidelines.

🔬

Science Claim Verification

Any physiology or nutrition science claim requires a web search and a real citation before it reaches me. No asserting from memory on empirical questions.

🌱

Progressive Enhancement

Core coaching first, then skills, then automation, then audit cadences. Each layer only got added when a real recurring need showed up — not because I thought it might be useful someday.

🔎

Skill Migration Detection

Every two weeks, the audit skill scans for manual workflows that have become recurring habits and flags them as candidates for automation. It tells me when it's ready to do more.

🚦

Structural Change Gate

Before I add anything new, a 3-point check has to pass: fits within the token budget? manually triggered? solves a real recurring problem? All three or it doesn't get built.
